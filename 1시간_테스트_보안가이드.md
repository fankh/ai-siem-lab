## 0. tools 컨테이너 접속 (테스트 실행 환경)

테스트·보안 명령은 **`tools` 컨테이너 안에서 실행**하면 호스트 OS(Windows/Mac/Linux)에 관계없이
동일하게 동작합니다(curl·jq·python3·nmap 내장). 호스트엔 **Docker 만** 있으면 됩니다.

### 0-1. 컨테이너 확인 / 기동
```bash
cd ai-siem-lab/labs/opensearch-ai
docker compose ps tools          # STATUS = Up 확인
docker compose up -d tools       # 안 떠 있으면 기동
```

### 0-2. 대화형 셸로 접속 (권장)
```bash
docker compose exec tools bash
#  → 프롬프트가 컨테이너 내부로 전환됨. 작업 디렉터리는 /lab (랩 폴더 마운트)
#  이후 명령은 localhost 대신 서비스명 사용:
curl -s http://opensearch:9200/_cluster/health | jq .status
exit                              # 컨테이너에서 빠져나오기
```
> `docker compose` 컨텍스트 밖이면 직접 접속도 가능: `docker exec -it ai-siem-tools bash`

### 0-3. 단발 명령 (셸 진입 없이)
```bash
docker compose exec tools curl -s http://opensearch:9200/security-web/_count | jq .count
docker compose exec tools python3 sample-data/load_sample_logs.py http://opensearch:9200
docker compose exec tools nmap -p 9200,5601,11434 opensearch ollama
```

### 접속 후 핵심
| 항목 | 값 |
|------|-----|
| 내부 주소 | OpenSearch `http://opensearch:9200` · Ollama `http://ollama:11434` (※ `localhost` 아님) |
| 작업 폴더 | `/lab` (= `labs/opensearch-ai`, `sample-data/` 포함) |
| 내장 도구 | curl · jq · python3 · nmap · dig · nc |

> 이후 `curl`/`python3`/`nmap` 같은 API·도구 명령은 컨테이너에서 실행합니다(주소 `opensearch:9200`/`ollama:11434`).
> `docker compose …` 관리 명령은 **호스트**에서 실행하세요. 호스트에서 API 직접 실행 시 서비스명→`localhost` 로 바꿉니다.

## 2. 검증(테스트) 가이드

### 2-1. 스택 기동 확인
```bash
cd ai-siem-lab/labs/opensearch-ai
docker compose ps
```
**기대 결과**: `opensearch`, `opensearch-dashboards`, `ollama` 3개가 `Up` / `healthy`.
실패 시: `docker compose logs opensearch | tail -50` 으로 원인 확인 (대개 메모리 부족).

### 2-2. 엔드포인트 헬스 체크
```bash
curl -s http://opensearch:9200/_cluster/health | jq '{status, number_of_nodes}'
curl -s -o /dev/null -w "dashboards=%{http_code}\n" http://opensearch-dashboards:5601/api/status
curl -s http://ollama:11434/api/version
```
**기대 결과**: OpenSearch `status: green|yellow`(단일노드는 yellow 정상) · dashboards `200` · Ollama 버전 출력.

### 2-3. 데이터 적재 검증
```bash
python3 sample-data/load_sample_logs.py http://opensearch:9200   # 컨테이너 /lab 기준
curl -s http://opensearch:9200/security-web/_count | jq .count
```
**기대 결과**: `716` (정상 550 + 브루트포스 160 + SQLi/XSS 6). 필드 확인:
```bash
curl -s "http://opensearch:9200/security-web/_search?size=1" | jq '.hits.hits[0]._source | keys'
```
→ `@timestamp, event, http, source, url, user_agent` 포함.

## 3. 보안 테스트 (스택 하드닝 점검)

> 실습 스택은 **보안 플러그인이 비활성(TLS·인증 없음)** 입니다. 아래로 "위험 노출"을 직접 확인하고
> 운영 시 조치를 이해합니다.

### 3-1. 인증 없이 접근되는지 (의도된 취약 상태)
```bash
curl -s http://opensearch:9200/_cat/indices?v
```
**관찰**: 인증 없이 인덱스 목록이 노출됨 → **실습 전용**. 운영에서는 보안 플러그인 활성 + 계정 필요.

### 3-2. 노출 포트 점검
```bash
docker compose ps --format '{{.Service}} -> {{.Ports}}'
```
**관찰**: `0.0.0.0:9200/5601/11434` 로 바인딩 → 외부에서 접근 가능.
**운영 권고**: `127.0.0.1:` 바인딩 또는 방화벽, 역방향 프록시 + 인증.

### 3-3. 이미지 출처/태그 점검
```bash
docker compose images
```
**관찰**: 공식 이미지(`opensearchproject/*`, `ollama/ollama`) + 고정 태그(`2.19.0`) 사용 여부.
**권고**: `latest` 지양, 다이제스트 고정, 취약점 스캔(`trivy image <이미지>`).

### 3-4. 운영 전환 체크리스트
- [ ] 보안 플러그인 활성 + TLS + 강한 admin 비밀번호
- [ ] 역할 기반 접근제어(RBAC), 최소 권한
- [ ] 포트 비공개 / 방화벽 / 감사 로깅
- [ ] 리소스 한도(메모리) 및 백업
